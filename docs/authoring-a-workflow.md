# Authoring a workflow

Use this checklist when filling in a stubbed workflow or adding a new one.

## 1. Decide command vs skill

| Use a command (`commands/<name>.md`) when… | Use a skill (`skills/<name>/SKILL.md`) when… |
| --- | --- |
| Workflow is purely conversational + writes text outputs | Workflow needs bundled helper scripts |
| All output can be generated by Claude inline | Workflow needs templates that get copied verbatim into the project |
| You want the simpler authoring model | You need richer frontmatter (`allowed-tools`, `context: fork`, etc.) |

Default to a command. Promote to a skill only when you actually need the extras.

## 2. Define the output contract

Before writing the command file, decide:

- **Output directory:** Always `product/supertools/<workflow>/`.
- **Output files:** Be specific. List every file the workflow writes. Match the Design OS conventions for output file formats where it makes sense.
- **What's deterministic vs. what's a decision:** Decisions get written to a `decisions.md` file at the root of the workflow's output dir. Generated assets live next to it.

## 3. Define prerequisites

What does this workflow need to exist before it can run?

- **Mandatory first check (every workflow):** Verify Design OS is complete by checking that `product-plan/README.md` and `product-plan/product-overview.md` both exist. If either is missing, tell the user to finish Design OS and stop. Use the exact prerequisite-check block from any existing workflow command — do not invent your own wording. This rule is non-negotiable; see `docs/architecture.md` ("Hard prerequisite").
- After the gate, list any additional Design OS files this workflow depends on (e.g. specific sections, design tokens).
- List any other supertools-design workflows that must run first.

If anything's missing, name the exact command to run and `STOP`. Don't half-run.

## 4. Pick the opinionated defaults

This is where most of the value lives. For each decision the workflow makes, pick a default:

- Note the decision in the command file ("By default we use X because Y").
- Only ask the user if the default genuinely doesn't apply.
- Document trade-offs concisely so the user can override when needed.

## 5. Follow the six-step skeleton

Copy the skeleton from `docs/architecture.md` (the "shape of a workflow command" section). Don't deviate without reason — consistency makes the whole system learnable.

## 6. Update the status file

Every workflow command must, before exiting:

- Mark its checkbox `[x]` (or `[~]` if partial) in `product/supertools/status.md`.
- Update the timestamp.
- Append a 1–2 line note under `## Notes` summarizing key decisions or deferrals.
- Update the `## Current step` line to the next recommended workflow command.

A snippet for the update step (in command-file pseudocode):

> Read `product/supertools/status.md`. Replace the line starting with `- [ ] **<workflow>**` with `- [x] **<workflow>**` (or `[~]` if partial). Replace the `_Last updated:_` timestamp with today's date. Append to the `## Notes` section: `- <YYYY-MM-DD> <workflow>: <one-line summary>`. Update the `## Current step` paragraph to point at the next recommended command.

## 7. Update the README and status template

If you're adding a new workflow (not just filling in an existing stub):

- Add it to the table in `README.md`.
- Add a checkbox line for it to the initial status template in `commands/start.md`.
- Add an entry to the "Workflows" section in `docs/architecture.md`.

## 8. Test the redo path

Run the command twice in the same project. The second run should detect the existing output and offer redo / refine / skip — not blindly overwrite. Match the Design OS pattern.
